# 项目技能与规则管理器

## 描述

项目级技能与规则管理器，负责为新项目或项目更新时安装、替换、升级、卸载技能（Skills）和规则（Rules）。核心目标是整合优质开源技能项目，选取项目适用的技能，配置合适的规则，避免不必要的 skills 占位和额外 token 消耗。

## 使用场景

- 新项目初始化时安装技能和规则
- 项目更新时同步/升级技能和规则
- 迁移项目中分散的技能到统一目录
- 根据项目技术栈自动选择适用技能

---

## 规则文档

技能调度和生命周期规则通过独立规则文件管理，采用按需加载机制：

| 规则文件 | 触发场景 |
|---------|---------|
| `skill-lifecycle-rules.md` | 安装/更新/升级/卸载技能和规则 |
| `skill-scheduling-rules.md` | 技能执行任务时 |
| `changelog-rules.md` | 更新更新日志为 vx.x.x |

## 核心流程

### 安装技能与规则

当用户说"安装技能和规则"或类似指令时，执行以下流程：

**第一步：识别项目中现有技能位置**

扫描项目中所有技能目录，识别现有技能：

**优先识别的目录：**
- `.agents/skills/` （目标目录，已安装的技能）
- `.trae/skills/` （旧目录，可能有需迁移的技能）
- `skills/` （常见技能目录）
- `.skills/` （常见技能目录）
- `docs/skills/` （文档目录下的技能）
- 任何包含 `SKILL.md` 或 `skill.md` 文件的目录

**识别方法：**
1. 使用 glob 搜索 `**/SKILL.md` 和 `**/skill.md`
2. 检查已知技能目录是否存在
3. 记录每个发现的技能名称、位置、内容摘要

**注意：** GitNexus、codegraph 等自管理工具的指定目录不在迁移范围内，扫描时记录但不迁移。

**第二步：整合技能项目**

整合以下技能项目：
- `https://github.com/affaan-m/ECC`
- `https://github.com/abhigyanpatwari/GitNexus`
- `https://github.com/colbymchenry/codegraph`
- `https://github.com/obra/superpowers`
- `https://github.com/multica-ai/andrej-karpathy-skills`

**第三步：前端设计技能评估**

评估以下技能是否可用于当前项目：
- `https://github.com/pbakaus/impeccable`
- `https://github.com/Leonxlnx/taste-skill`

判断标准：项目技术栈匹配度、功能契合度、复杂度评估

**第四步：技术栈适配筛选**

| 项目类型 | 优先技能 | 筛选标准 |
|---------|---------|---------|
| Node/前端项目 | 前端设计、代码规范、测试技能 | package.json 检测框架 |
| Python 项目 | 数据处理、ML 工程、API 技能 | requirements.txt / pyproject.toml |
| Rust 项目 | 内存安全、并发、系统编程技能 | Cargo.toml |
| Go 项目 | 微服务、网络、容器化技能 | go.mod |
| 全栈项目 | 综合技能包 | 同时满足前后端标准 |

**第五步：技能冲突处理**

| 冲突类型 | 处理方式 |
|---------|---------|
| 同名不同版本 | 询问用户：保留旧版/安装新版/同时保留（重命名） |
| 功能重叠 | 询问用户选择其中一个 |
| 依赖冲突 | 自动选择兼容版本，或报告给用户解决 |

**第六步：技能迁移与安装**

- **自管理技能（不迁移）**：GitNexus、codegraph 等工具的指定目录
- **可迁移技能**：移动到 `.agents/skills/` 目录

**第七步：清理原技能目录**

删除已被清空的原技能目录，保留自管理工具目录和其他非技能文件。

**第八步：规则文档配置**

创建 `.agents/rules/` 目录，生成以下规则文件：
- `skill-lifecycle-rules.md` - 技能生命周期规则
- `skill-scheduling-rules.md` - 技能调度规则
- `changelog-rules.md` - 更新日志规则
- `project-rules.md` - 项目通用规则（如有需要）

**第九步：MCP 改 CLI**

| 优先级 | 策略 |
|-------|------|
| 1 | 官方 CLI |
| 2 | 社区 CLI 包装器 |
| 3 | HTTP API 替代 |
| 4 | 功能等效 CLI |

**第十步：生成技能清单**

生成 `.agents/skills/INDEX.md` 记录已安装技能列表、用途、优先级、依赖工具。

## 技能调度规则

技能执行任务时遵循以下规则：

**难度分级：**
| 级别 | 难度 | 典型任务 | 最大并行数 |
|------|------|---------|-----------|
| L1 | 简单 | 翻译、格式化、简单修复 | 1 |
| L2 | 中等 | 功能实现、单元测试 | 2 |
| L3 | 复杂 | 架构设计、API 设计 | 3 |
| L4 | 专家 | 性能优化、安全审计 | 5 |

**调度原则：**
1. 项目内技能优先于全局技能
2. 按需加载，避免全量加载
3. 单一职责，复杂任务多技能协作
4. 串行优先，避免并行冲突
5. 多技能执行后必须结果汇总

**场景映射：**
- 开发场景：写新功能、重构代码、写测试、代码审查
- 调试场景：Bug 定位、性能问题、崩溃分析
- 部署场景：构建发布、环境配置、CI/CD
- 文档场景：生成文档、更新日志

## 更新日志规则

**触发指令：** 用户说"更新更新日志为 vx.x.x"

**流程：**
1. 收集上一个 tag 到最新提交的所有 commit
2. 按类型分类：🎉新增/✨优化/🔧修复/⚡调整/🏗️重构/📊文档/🎨样式
3. 生成 CHANGELOG 内容
4. 更新所有版本号文件

---

**开始执行。等待用户指令，选择合适的技能和规则为项目赋能。**
