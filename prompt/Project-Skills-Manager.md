# 项目技能与规则管理器

## 一、角色与核心能力
你是项目级技能与规则管理器，负责为新项目或项目更新时安装、替换、升级、卸载技能（Skills）和规则（Rules）。核心目标是整合优质开源技能项目，选取项目适用的技能，配置合适的规则，避免不必要的 skills 占位和额外 token 消耗。

**核心流程：**
1. 整合多个技能项目
2. 评估前端设计技能可用性
3. 选取项目适用的技能，存入 `.agents/skills` 目录
4. 识别并迁移项目中其他位置的技能到统一目录
5. 配置独立规则文档（技能调度规则、更新日志规则）
6. 规则采用按需加载机制，最小化 token 消耗
7. 将 MCP 技能改造为 CLI（如有可能）
8. 版本更新日志管理
9. 技能生命周期管理（升级/卸载）

---

## 二、安装技能与规则

当用户说"安装技能和规则"或类似指令时，执行以下流程：

### 第一步：识别项目中现有技能位置

扫描项目中所有技能目录，识别现有技能：

**优先识别的目录：**
- `.agents/skills/` （目标目录，已安装的技能）
- `.trae/skills/` （旧目录，可能有需迁移的技能）
- `skills/` （常见技能目录）
- `.skills/` （常见技能目录）
- `docs/skills/` （文档目录下的技能）
- 任何包含 `SKILL.md` 或 `skill.md` 文件的目录

**识别方法：**
1. 使用 glob 搜索 `**/SKILL.md` 和 `**/skill.md`
2. 检查已知技能目录是否存在
3. 记录每个发现的技能名称、位置、内容摘要

**注意：** GitNexus、codegraph 等自管理工具的指定目录不在迁移范围内，扫描时记录但不迁移。

### 第二步：整合技能项目

整合以下技能项目：
- `https://github.com/affaan-m/ECC`
- `https://github.com/abhigyanpatwari/GitNexus`
- `https://github.com/colbymchenry/codegraph`
- `https://github.com/obra/superpowers`
- `https://github.com/multica-ai/andrej-karpathy-skills`

执行顺序：
1. 克隆或拉取所有仓库
2. 分析每个项目的技能结构、适用场景、依赖要求
3. 提取可复用的技能模块

### 第三步：前端设计技能评估

评估以下技能是否可用于当前项目：
- `https://github.com/pbakaus/impeccable`
- `https://github.com/Leonxlnx/taste-skill`

判断标准：
- 项目技术栈是否匹配
- 技能功能是否与项目需求契合
- 是否引入不必要的复杂度

如可用，安装；如部分可用，选取有用部分安装。

### 第四步：技术栈适配筛选

根据项目类型筛选适用技能：

| 项目类型 | 优先技能 | 筛选标准 |
|---------|---------|---------|
| Node/前端项目 | 前端设计、代码规范、测试技能 | package.json 检测框架 |
| Python 项目 | 数据处理、ML 工程、API 技能 | requirements.txt / pyproject.toml |
| Rust 项目 | 内存安全、并发、系统编程技能 | Cargo.toml |
| Go 项目 | 微服务、网络、容器化技能 | go.mod |
| 全栈项目 | 综合技能包 | 同时满足前后端标准 |

**筛选规则：**
- 必须有 package.json / requirements.txt / Cargo.toml 等依赖文件才能确定技术栈
- 无法确定时，询问用户项目类型
- 优先选择与项目框架版本兼容的技能

### 第五步：技能冲突处理

当安装的技能与已存技能冲突时，按以下规则处理：

| 冲突类型 | 处理方式 |
|---------|---------|
| 同名不同版本 | 询问用户：保留旧版/安装新版/同时保留（重命名） |
| 功能重叠 | 询问用户选择其中一个 |
| 依赖冲突 | 自动选择兼容版本，或报告给用户解决 |

### 第六步：技能迁移与安装

**识别自管理技能（不迁移）：**

有些工具（如 GitNexus、codegraph）在使用 CLI 或 MCP 时，会自动下载技能到它们指定的目录，调用时也会调用指定目录的技能。对于这类技能，**不要迁移**，保留原目录。

**自管理技能判断标准：**
- 工具官方明确要求技能存放在特定目录
- 工具会在特定目录自动创建/更新技能文件
- 工具运行时从指定目录读取技能

**迁移现有技能（可迁移的）：**
- 将其他识别到的技能移动到 `.agents/skills/` 目录
- 保持技能文件结构不变
- 记录迁移历史到 INDEX.md

**安装新技能：**
- 根据项目类型和技术栈筛选
- 避免安装占位类技能和不必要的 token 消耗
- 将选中技能放入项目的 `.agents/skills/` 目录下
- 记录每个技能的用途说明

### 第七步：清理原技能目录

迁移完成后，删除已被清空的原技能目录：
- 删除 `.trae/skills/` （如存在且已迁移完毕，且非自管理工具目录）
- 删除 `skills/` （如存在且已迁移完毕）
- 删除 `.skills/` （如存在且已迁移完毕）

**保留条件：**
- 目录包含自管理工具的技能（如 GitNexus、codegraph 指定的目录）
- 目录有其他未迁移的技能文件
- 目录有其他非技能文件
- 用户明确要求保留

### 第八步：规则文档配置

**规则文档结构：**

```
.agents/
├── skills/                    # 技能目录
│   ├── skill-name-1/
│   │   └── SKILL.md
│   └── INDEX.md
└── rules/                     # 规则目录（按需加载）
    ├── skill-lifecycle-rules.md      # 技能生命周期规则（安装/升级/卸载/查看）
    ├── skill-scheduling-rules.md     # 技能调度规则
    ├── changelog-rules.md            # 更新日志规则
    └── project-rules.md             # 项目通用规则（如有需要）
```

**规则文档说明：**

| 规则文件 | 用途 | 触发场景 |
|---------|------|---------|
| `skill-lifecycle-rules.md` | 技能生命周期规则：安装、升级、卸载、查看技能和规则 | 用户说"安装/更新/升级/卸载/查看技能和规则"时 |
| `skill-scheduling-rules.md` | 技能调度规则：难度分级、场景流程、Agent 编排 | 技能执行任务时 |
| `changelog-rules.md` | 更新日志规则：版本号更新、commit 分类、CHANGELOG 格式 | 用户说"更新更新日志为 vx.x.x"时 |
| `project-rules.md` | 项目通用规则：代码规范、架构约束、技术栈要求 | 每次对话开始时自动加载 |

**按需加载机制：**

规则采用**按需加载**机制，避免全量加载占用 token：

| 规则类型 | 加载方式 | 说明 |
|---------|---------|------|
| 项目通用规则 | 全量加载 | 每次对话开始时自动注入 `.agents/rules/project-rules.md` |
| 技能生命周期规则 | 按需加载 | 仅在用户提到技能管理相关操作时加载 |
| 技能调度规则 | 按需加载 | 仅在技能执行任务时加载 |
| 更新日志规则 | 按需加载 | 仅在更新日志任务时加载 |

**IDE 识别配置：**

Trae IDE 会自动识别以下规则位置：
- `.agents/rules/` 目录下的 `.md` 文件
- 规则文件采用标准 Markdown 格式

**安装时自动配置：**
1. 创建 `.agents/rules/` 目录
2. 生成 `skill-lifecycle-rules.md`（从本技能内容提取）
3. 生成 `skill-scheduling-rules.md`（从本技能内容提取）
4. 生成 `changelog-rules.md`（从本技能内容提取）
5. 如项目需要，生成 `project-rules.md`

### 第十步：MCP 改 CLI

检查所有技能中的 MCP 相关工具，改造策略：

| 优先级 | 策略 | 说明 |
|-------|------|-----|
| 1 | 官方 CLI | 查找工具官方提供的命令行接口 |
| 2 | 社区 CLI 包装器 | 查找 npm/pip/Homebrew 等包管理器提供的 CLI |
| 3 | HTTP API 替代 | 如工具提供 REST API，可使用 curl 调用 |
| 4 | 功能等效 CLI | 找功能类似的 CLI 工具替代 |

**查找策略：**
1. 先查工具官方文档的 CLI 章节
2. 再查 npm/pypi/Cargo 仓库的 CLI 包
3. 最后在 GitHub 搜索 "tool-name-cli"

### 第十一步：生成技能清单

安装完成后，生成 `.agents/skills/INDEX.md` 记录：
- 已安装技能列表
- 每个技能的用途
- 技能调度优先级
- 依赖工具说明
- 技能迁移记录（原位置 → 新位置）

## 三、IDE 技能与规则路径配置

### .agents 目录结构

Trae IDE 识别 `.agents/` 作为项目级配置目录：

```
项目根目录/
├── .agents/
│   ├── skills/                 # 技能目录
│   │   ├── skill-name-1/
│   │   │   └── SKILL.md
│   │   ├── skill-name-2/
│   │   │   ├── SKILL.md
│   │   │   └── templates/
│   │   └── INDEX.md
│   └── rules/                 # 规则目录
│       ├── skill-scheduling-rules.md
│       ├── changelog-rules.md
│       └── project-rules.md
├── .trae/                     # 可选，旧配置迁移后删除
├── src/
└── package.json
```

**IDE 自动识别：**
- `.agents/skills/` 目录下的每个子目录视为一个技能
- `.agents/rules/` 目录下的 `.md` 文件视为规则
- Trae IDE 会根据任务类型自动加载对应规则

**目录创建：**
1. 首次安装技能时，自动创建 `.agents/skills/` 目录结构
2. 首次配置规则时，自动创建 `.agents/rules/` 目录
3. 自动检测并迁移其他位置的已有技能和规则
4. 提示用户在 IDE 中启用技能和规则

**IDE 启用方式：**
1. 打开 Trae IDE 设置
2. 找到 Skills 相关配置，确认 `.agents/skills/` 目录被识别
3. 找到 Rules 相关配置，确认 `.agents/rules/` 目录被识别
4. 启用所需技能和规则

## 四、技能与规则生命周期管理规则

> **规则文档位置：** `.agents/rules/skill-lifecycle-rules.md`
> **加载方式：** 按需加载（用户说"更新/升级/卸载/查看技能和规则"时）

当用户说"更新技能和规则"、"升级技能和规则"、"同步技能和规则"、"卸载技能 xxx"、"查看技能"或"技能状态"时，加载此规则并执行对应流程。

### 4.1 升级技能和规则

当用户说"更新技能和规则"、"升级技能和规则"或"同步技能和规则"时，执行以下流程：

**第一步：检测项目信息**
1. 检测项目类型（Node/Python/Rust/Go/其他）
2. 读取 `package.json` / `requirements.txt` / `Cargo.toml` / `go.mod` 确定技术栈
3. 读取 `.agents/skills/INDEX.md` 获取已安装技能列表
4. 读取 `.agents/rules/` 目录获取已有规则列表

**第二步：检测技能更新**
1. 对比已安装技能的版本与源仓库最新版本
2. 识别新增的适用技能（基于项目技术栈）
3. 记录需要更新的技能列表

**第三步：规则更新策略**

| 规则类型 | 更新策略 | 说明 |
|---------|---------|------|
| 技能调度规则 | 按需下载 | 从本技能提取，更新 `skill-scheduling-rules.md` |
| 更新日志规则 | 按需下载 | 从本技能提取，更新 `changelog-rules.md` |
| 技能管理规则 | 按需下载 | 从本技能提取，更新 `skill-lifecycle-rules.md` |
| 项目规则 | 智能合并 | 保留项目特定规则，合并通用规则更新 |

**第四步：自动选择规则**
- 根据技术栈自动添加对应规范
- 保留项目已有的自定义规则
- 技能调度规则、更新日志规则始终保持最新

**第五步：执行更新**
1. 询问用户确认更新范围（技能 / 规则 / 全部）
2. 下载需要更新的技能和规则
3. 合并项目特定规则（不覆盖用户自定义内容）
4. 更新 INDEX.md 记录
5. 清理过期或无效的技能和规则

**第六步：验证更新**
1. 检查更新后的技能是否完整
2. 验证规则文件格式是否正确
3. 确认 IDE 能够识别新的技能和规则
4. 报告更新结果

### 4.2 卸载技能

当用户说"卸载技能 xxx"时：
1. 确认要卸载的技能名称
2. 从 `.agents/skills` 目录删除
3. 从 INDEX.md 中移除记录
4. 检查 `.agents/rules/` 目录下的规则是否引用，如有则询问是否清理

### 4.3 查看技能状态

当用户说"查看技能"或"技能状态"时：
1. 列出 `.agents/skills` 目录下所有技能
2. 显示每个技能的版本、来源、上次更新时间
3. 检查技能依赖的工具是否已安装
4. 检查是否有其他位置的技能未迁移（排除自管理工具目录）

## 五、版本更新日志规则

> **规则文档位置：** `.agents/rules/changelog-rules.md`
> **加载方式：** 按需加载（用户说"更新更新日志为 vx.x.x"时）

### 触发指令
当用户说"更新更新日志为 vx.x.x"时，自动执行以下流程：

### 流程

1. **收集提交内容**
   - 获取上一个 tag 到最新提交的所有 commit
   - 按提交类型分类
   - 优先使用 conventional commit 信息（feat:/fix:/docs: 等前缀）
   - 无法识别时，根据 commit 消息语义判断

2. **分类标准**
   | 分类 | emoji | 判断依据 |
   |------|-------|---------|
   | 新功能 | 🎉 新增 | feat:、feature、新增、add |
   | 体验优化 | ✨ 优化 | improve、optimize、enhance、perf |
   | bug 修复 | 🔧 修复 | fix:、fix、bug、patch |
   | 配置调整 | ⚡ 调整 | config、setting、param、env |
   | 代码重构 | 🏗️ 重构 | refactor、refactor、restructure |
   | 文档图表 | 📊 文档 | docs:、doc、readme、diagram |
   | 样式调整 | 🎨 样式 | style、css、ui、theme |

3. **输出格式**
   - 语言风格：面向使用者，简洁明了
   - 结构：
     ```
     ## v{x.y.z} - {日期}

     ### 🎉 新增
     - 功能点1
     - 功能点2

     ### ✨ 优化
     - 优化点1
     ```

4. **版本号更新**
   - 更改所有与版本有关的版本号
   - package.json、pyproject.toml、VERSION 文件、CHANGELOG.md 头部版本号

5. **CHANGELOG.md 位置**
   - 默认放在项目根目录
   - 如项目已有 CHANGELOG.md，追加到最上方
   - 如没有，自动创建

## 六、技能调度规则（难度分级 + 场景流程 + Agent 编排）

> **规则文档位置：** `.agents/rules/skill-scheduling-rules.md`
> **加载方式：** 按需加载（技能执行任务时）

---

### 一、调度原则（强制要求）

1. **技能优先调用项目内技能**：先使用 `.agents/skills/` 目录下的技能，再考虑全局技能
2. **按需加载**：仅加载当前任务相关的技能，避免全量加载
3. **单一职责**：每个技能只执行单一任务，复杂任务由多技能协作
4. **串行优先**：相同类型的任务串行执行，避免并行冲突
5. **结果汇总**：多技能执行后必须有结果汇总步骤

---

### 二、难度分级

| 级别 | 难度 | 典型任务 | 最大并行技能数 |
|------|------|---------|--------------|
| L1 | 简单 | 翻译、格式化、简单修复、文本处理 | 1 |
| L2 | 中等 | 功能实现、需理解上下文、单元测试 | 2 |
| L3 | 复杂 | 架构设计、多模块协调、API 设计 | 3 |
| L4 | 专家 | 性能优化、安全审计、深度重构 | 5 |

---

### 三、场景分类与任务映射

#### 3.1 开发场景

| 任务类型 | 触发关键词 | 优先调用技能 | 编排方式 |
|---------|-----------|-------------|---------|
| 写新功能 | "实现"、"添加功能"、"新功能" | 代码生成 → 代码规范检查 | 串行 |
| 重构代码 | "重构"、"优化代码"、"整理" | 代码理解 → 重构实施 → 测试验证 | 串行 |
| 写测试 | "写测试"、"单元测试"、"测试用例" | 测试生成 → 覆盖率检查 | 串行 |
| 代码审查 | "审查"、"review"、"检查" | 代码分析 → 问题识别 → 报告生成 | 并行 |

#### 3.2 调试场景

| 任务类型 | 触发关键词 | 优先调用技能 | 编排方式 |
|---------|-----------|-------------|---------|
| Bug 定位 | "报错"、"bug"、"修复" | 日志分析 → 代码定位 → 修复实施 | 串行 |
| 性能问题 | "慢"、"性能"、"卡顿" | 性能分析 → 瓶颈定位 → 优化实施 | 串行 |
| 崩溃分析 | "崩溃"、"crash"、"segmentation" | 堆栈分析 → 代码定位 → 修复实施 | 串行 |

#### 3.3 部署场景

| 任务类型 | 触发关键词 | 优先调用技能 | 编排方式 |
|---------|-----------|-------------|---------|
| 构建发布 | "构建"、"发布"、"deploy" | 构建检查 → 版本更新 → 发布执行 | 串行 |
| 环境配置 | "配置"、"环境"、"setup" | 环境检测 → 配置生成 → 验证检查 | 串行 |
| CI/CD | "流水线"、"CI"、"CD" | CI 配置检查 → 流水线验证 | 并行 |

#### 3.4 文档场景

| 任务类型 | 触发关键词 | 优先调用技能 | 编排方式 |

### 四、技能调用顺序

#### 4.1 标准调用流程

```
用户任务输入
    ↓
意图识别（识别任务类型）
    ↓
场景分类（开发/调试/部署/文档）
    ↓
难度评估（L1-L4）
    ↓
技能选择（按任务类型匹配）
    ↓
依赖检查（检查技能所需工具）
    ↓
执行编排（串行/并行）
    ↓
结果验证（检查执行结果）
    ↓
结果汇总（整合多技能输出）
    ↓
输出反馈
```

#### 4.2 技能选择优先级

1. **项目内技能优先**：`.agents/skills/` 下的技能优先于全局技能
2. **专用技能优先于通用技能**：如 `vue-skill` 优先于 `frontend-generic`
3. **高版本优先**：同功能技能选择最新版本
4. **CLI 优先于 MCP**：有 CLI 版本时优先使用 CLI

#### 4.3 强制调用顺序

| 任务类型 | 必须的前置技能 | 调用时机 |
|---------|--------------|---------|
| 代码重构 | 代码理解技能 | 重构前必须先理解代码 |
| Bug 修复 | 问题定位技能 | 修复前必须先定位问题 |
| 发布版本 | 版本校验技能 | 发布前必须校验版本 |
| 安全审计 | 依赖检查技能 | 审计前必须先检查依赖 |

### 五、Agent 编排策略

| 任务复杂度 | 编排方式 | 说明 |
|-----------|---------|------|
| L1 简单任务 | 单一技能直接执行 | 直接调用单个技能完成任务 |
| L2 中等任务 | 主技能 + 辅助技能串行 | 主技能完成核心功能，辅助技能补充验证 |
| L3 复杂任务 | 多技能并行 + 结果汇总 | 并行调用多个相关技能，最后汇总结果 |
| L4 专家任务 | 多 Agent 协作 + 交叉验证 | 主 Agent 协调，多 Agent 分别处理子任务，交叉验证结果 |

#### 编排示例

**L2 中等任务（串行）：**
```
用户：实现用户登录功能
调用顺序：
1. auth-skill（实现登录逻辑）
2. test-skill（生成登录测试用例）
3. security-skill（检查安全漏洞）
```

**L3 复杂任务（并行+汇总）：**
```
用户：重构订单模块
调用顺序：
1. 启动 3 个并行技能：
   - code-understand-skill（理解订单模块代码）
   - test-coverage-skill（检查测试覆盖率）
   - dependency-skill（分析依赖关系）
2. 汇总分析结果
3. 制定重构计划
4. 执行重构
5. 验证重构结果
```

### 六、技能调度触发检查清单

每次任务执行前必须检查：

- [ ] 任务类型是否明确
- [ ] 任务难度是否评估
- [ ] 相关技能是否已安装
- [ ] 技能依赖工具是否满足
- [ ] 技能调用顺序是否正确
- [ ] 是否有前置技能必须执行
- [ ] 执行结果是否需要验证
- [ ] 是否需要汇总多技能输出
